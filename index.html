多包管理

发表于 2020-12-07

多包管理方式

常见的开发方式可以将多个项目放在同一个git仓库里，采用不同的子目录做区分即可，一撸到底，但是不同的包管理起来相对比较混乱，一个依赖可能会造成全局安装，或者不同的项目会相互影响，如果设计到子项目之间的相互调用就不太优雅。

采用monorepo的多包管理方式，即还是将多个项目放在同一个git仓库里

采用multirepo的多包管理方式，将多个项目放在不同的git仓库里

相比较两种多包管理方式，monorepo的方式有两个优点

  • 项目内某一个包更新，可以在内部直接更新引用即可，如果是多git仓库管理等，则需要发布此包，另外引用到该包的包也都需要再发布一次

  • 可以在项目最外层定义一些公共配置，比如eslintrc.js等

使用lerna

lernaJS是由Babel团队出的一个多包管理工具。因为Babel包含很多子包，以前都是放在多个仓库里的，管理比较困难，特别是有调用系统内包的时候，发布比较麻烦。所以为了能更好更快的夸包管理，babel推出了lernaJS，使用了monorepo的概念，现在React,Babel,Angular,Jest都在使用这个工具来管理包。

基本使用方式

  • 初始化lerna项目
  • 创建子项目demo1
  • 创建子包依赖
  • 安装外部依赖

  • 其他命令

其他用法可以结合lerna exec比如删除某个子项目的包leran exec 'yarn remove lodash' --scope=utils

lerna的版本lerna.json中的version配置项

  • Fixed

全局固定为一个版本，发包的时候全部统一升级一个版本

当升级时，遵循semver规范

  • Independent

每个子包使用各自的版本控制

lerna拓扑执行

这一点也是lerna一大卖点，相对比与yarn的多包管理方式，他可以分析包中的依赖然后去打包执行

比如上面提到的demo1项目依赖到了utils项目，那么打包时，就应该先打包utils包，再去打包demo1包

postMessage基本使用

  • otherWindow: 其他窗口的一个引用，比如 iframe 的 contentWindow 属性,iframe的父窗口window.parent,或者window.open返回的窗口对象等。
  • message 要发送的消息，它将会被结构化克隆算法序列化,也可直接使用JSON.stringify传递string
  • targetOrigin 要接收message的窗口，可理解otherWindow正确的URL，可以为一个具体的值http://www.baidu.com,或者为*即认为均可向otherWindow发送消息，无论otherWindow的地址是怎样的
  • transfer 是一串和message 同时传递的 Transferable 对象. 这些对象的所有权将被转移给消息的接收方，而发送一方将不再保有所有权。

前端性能监控

发表于 2020-11-16 更新于 2020-11-17

监控相关性能指标

  • FP first paint 首次绘制
  • FCP first contentful paint 首次内容绘制（有东西出来就行）
  • FMP first meaningful paint 首次有意义的绘制（有大概意义即可）

FMP

对fmp的定义不尽相同，什么是有意义的对于不同的业务方理解是不同的 如

  • 博客：标题，博文显示出来即可
  • 电商网站，更多的图片被显示出来
  • 搜索引擎，搜索内容显示出来

google提出了一个较为通用的解决方案，就是页面上DOM变动最大时，即认为做出了首次有意义的绘制，同时要考虑一些内容页面（页面内容超出显示器内容时的情况，页面内容/屏幕内容），以及引用到了特殊的字体时，要考虑字体没有加载出来，从而加载出的内容也是没有太大的意义的。

使用浏览器相关API获取数据

浏览器原生的performanceAPI记录数据

  • performance.navigation 页面加载还是刷新，发生了多少次重定向

  • performance.timing 页面加载的各阶段时长，其中可以获得页面各个性能指标的时间

  • performance.memory 内存使用情况（chrome中支持的非标准特性）

  • performance.timeorigin 性能开始记录时高精度的时间戳

获取指标数据

  • performance.now() 获取当前页面开始加载到现在的时间戳（毫秒），而不是1970年的那个时间戳

  • perfromance.getEntries() 获得各种打点记录数据

  • perfromance.getEntriesByName() 通过名称获取记录数据（自定义mark的数据也可以这样取得）

  • performance.getEntriesByType() 通过类型获取指标，比如paint, 绘制相关，FP,FCP等。resource各种资源的时间节点

  • performance.mark(<tag string>) 自定义打点，之后可通过getEntriesByName查询到

  • performance.clearMarks(<tag string>) 清除相关自定义打点监控数据

FPS

一般页面正常FPS维持在60会是一个比较理想的状态，如果页面降到20FPS后，会产生明显的卡顿感，由于毕竟浏览器还是要计算绘制的，一直让其保证60FPS也是不太可能的，比如在用户产生滑动时，页面FPS会突然下降，但是只要后续能够恢复正常就没有问题，所以我们在检测FPS一定是一段时间内的检测，设定相应的阈值，比如降低到某FPS，持续多少次，或者多久进行上报，或者采样每秒的FPS进行比对。

监控js报错

页面执行的错误也需要我们进行捕获，由于浏览器的跨域安全策略，部分script脚本加载时出错是捕获不到的.

监控js遇到的问题

出于安全考虑，浏览器会刻意隐藏其他域的JS文件抛出的具体错误信息，这样做可以有效避免敏感信息无意中被不受控制的第三方脚本捕获。因此，浏览器只允许同域下的脚本捕获具体错误信息，而其他脚本只知道发生了一个错误，但无法获知错误的具体内容。

总结

前端页面虽然在开发阶段，可以使用lighthouse或者webpagetest,但是真正在线上的情形我们是无从的得知的，需要真正统计线上的数据才能有一个大致的了解，就需要我们对线上的页面进行一些监控，从而有针对性的进行优化分析。

对于前端性能监控，总的来说关注两个大点

  • 页面加载时各种数据

用来计算页面的各种打开性能，打开快不快，页面加载的时间如何，用户体验好不好

  • 页面运行时的报错

线上出现的问题，我们是没法调试的，不可能让用户去抓包，所以及时上报一些异常情况会及时发现问题，对于第三方包要开启crossorigin="anonymous"，或者自定义try-catch包裹错误

  • 上报时机

由于毕竟监控也是使用js，或多或少还是会有一定的影响，这里要做一定的考量，以及使用一些监控方式（线上抽样检测，而不是全量检测），另外鉴于js单线程的特性，主线程阻塞时也没有办法继续监控，可以考虑采取service-worker进行向主进程推送心跳包的方式，判定主线程是否正常。

https加密

发表于 2020-11-05

简要介绍https是如何保证通信安全的

加密方式

  • 对称加密

通过一个密钥，加密又可解密，通常此类算法性能损耗较小，加密速度相对比较快（简单的加密方式）

  • 非对称加密

加密和解密不是同一个密钥。公钥加密，私钥解密；私钥加密，公钥解密。

公钥是可以公开的，私钥通常是自己保存的。

非对称加密的应用场景

加密通信

采用公钥加密数据，只有私钥可以解密，也就是只有私钥拥有者可以破译消息，就可以达到加密通信的要求

鉴权

采用私钥加密一定的身份信息，公钥解密后得到可信的身份数据，虽然攻击者也可以使用公钥解密出身份信息，但是没有办法使用私钥加密，所以只有私钥拥有者可以加密身份信息，之后正确的被对应公钥解开

消息完整性

对原始消息产生一个信息摘要（缩短加密内容），使用私钥进行加密，之后将加密信息摘要和消息一起发给对方，对方拿到公钥进行解密，并且使用相同算法产生摘要，和解密信息进行比对，攻击者由于没有私钥，就没有办法修改加密的信息摘要，从而保证了原始信息的完整性

网络通信的安全

网络传输包，很容易被第三方监听，我们要做的就是加密我们的数据，即用一个对称加密算法，可以大大提供安全性，但是如何将这个对称加密的密钥安全的送到通信方手中，是一个重要的问题。

解决的是，传送对称加密密钥的安全问题，（非对称加密也安全，但是大量通信会比较浪费性能）

颜色理论

颜色理论是一种不传输密钥，但是却可以让双方都得到密钥的方法，他的唯一理论就是不可逆的过程，比如一杯水是红色，你不知道之前是哪两种颜色勾兑起来的。

Alice和Bob最终得到了密钥颜色（棕色），但是这个过程中，他们俩自身的秘密颜色，都没有交换出来，只是交换了一次中间颜色，即使攻击人拿到后，也不知道是什么颜色相加得到

使用非对称加密交换密钥

在访问服务器时，我们只需要拿到服务器的公钥，然后加密一个密钥，之后服务器使用私钥解密，就安全拿到了对称加密的密钥。

但是我们需要拿到服务器的公钥，服务器的公钥是不能预先保存的，又到了如何保证拿到的公钥就是服务器的公钥，否则还是会造成攻击方透明的监听

实际拿到了攻击者的公钥，他用私钥解密出对称密钥，再用真的服务器公钥加密，转发给服务器，实际上他已经拿到了对称密钥

CA机构的公钥

为了打破这种死循环，有一些安全认证机构，直接将他们的公钥嵌入在操作系统中或者是浏览器内部，这样就保证了他们的公钥是真实可信的，直接从本地取就可以了，从而我们借助这个公钥可以进行鉴权，拿到后续真正的服务端公钥。

加密过程

  1. 客户端将自己的公钥交给CA，去进行一次加密，CA机构用自己的私钥加密证书内容
  • 服务端将公钥交给CA证书，可以采用CA的公钥加密，CA用私钥进行一次解密，确保服务端公钥传输CA安全
  • CA将加密的数据用自己的私钥加密，生成信息摘要，服务端拿到加密数据后，用公钥解密，对比一下加密的内容是否被篡改，进行数据完整性校验。确保拿到的东西是安全完整的。

CA私钥 + 服务端公钥 = CA私钥加密(服务端公钥)

  1. 客户端将这个加密内容传输给客户端，客户端使用本地的CA公钥进行一次解密，也就得到了可靠的服务端公钥。
  • 上一步，还会在证书中携带一些持有人相关信息，也就是用CA公钥解密出来的东西，还可以确定这串加密是不是从服务端发过来的，因为只有CA可以加密内容，所以不会有错

CA公钥 + CA私钥加密(服务端公钥) = 服务端公钥

  1. 客户端拿到服务端公钥后，以及一个加密算法，随机生成一个对称密钥，交给服务端

服务端公钥 + 对称密钥 = 服务端公钥加密(对称密钥)

  1. 服务端拿到自己的私钥解密这一串数据，得到最终加密的数据

服务端私钥 + 服务端公钥加密(对称密钥) = 对称密钥

!非空断言操作符

断言为非空用于过滤掉空值，只是在ts类型检测层面上做了不提示，代码本质并没有任何修改

经过ts的编译后，不会有任何处理

?.可选运算符

可选链，在遇到空值时，会立即停止表达式的执行，代码本质会做一些修改,当可选链中途断开的时候，会返回undefined

利用可选链，可以做一些深层次的选择

??空值合并运算符

当操作符左边的值为null或者为undefined时，会取右侧的值，否则会返回左侧的值/表达式的值

ts最终会经过一次编译

js中&&是判断falsy的值，比如'', 0, NaN三个falsy值，&&会认为失败,而??会认定为成功，也即??是为判断取值而诞生的

短路效果

可配合可选链，做去参数的默认值

由于?.操作取不到值时会返回undefined，符合??的判定依据

类型去除/增加空值

参数取默认值，不要加?

发布订阅与观察者

发表于 2020-10-18 更新于 2020-10-19

发布订阅与观察者模式总是被混淆，不容易分辨，其实这两种模式最初都是同一种，之后进行了一些发展与演变，才有了发布订阅这种解耦的方式，就自己的理解谈一谈这两种方式的具体区别。

两者的区别

发布订阅可以说是观察者模式的一种升级

观察者模式核心是我知道你的存在，当发生了变化时我直接通知大家发生的变化。

发布订阅模式核心是我不知道其他人是否存在，我需要告诉第三方有事件了，第三方负责去通知各个订阅人，同理各个订阅人也只是订阅了，具体会不会发生这个事件订阅人也不知道。

观察者模式是松耦合的，发布订阅是解耦合的。

解耦合当然是好事情了，但是还是需要去看具体的场景了，如果彻底解耦就需要第三方来帮助我们对接，那么无疑会增加我们的代码

如果是大型系统的对接，我们就需要使用解耦合的场景，未来我们也不知道会有那些人来使用，会有谁接入进来，所以需要一个第三方平台来管控，是比较便于后续系统的继续拓展的。

如果是我们自己内部代码，那么松耦合也是可以接受的，我们在编写代码就能掌控到全局，为了代码模块今后的拓展性，采用松耦合的方式也完全是ok的。

当发生取消操作时（不订阅），观察者模式主体是需要亲自操作的，而发布订阅模式发布者是不需要感知的。

Context都传递下去的问题

比如某个自组件中可能只是单纯想进行一个数据的修改，但是由于context中同时也依赖了state,所以当顶层组件的state发生改变时，该组件也会进行一次重新的渲染，那这样其实不符合我们的预期的，所以就将Context存储的数据拆分，那我们的核心就是读写分离！

可能会发生Context套娃，没关系先把主要问题解决了

将Context存储的数据抽离出来

目前是将所有的数据都存储在顶层组件中的，我们也可以使用一个自定义Hooks去存储我们的数据，最终附带吐出一个带有Context的组件即可

最终我们创建了一个类似store的组件，即CountContext,之后便可以方便的修改我们的数据，将此Context内部的数据都封装在自定义Hooks内部，也减少了对外部代码的侵入，使用起来比较简洁。

配合Reducer使用

前面一直使用useState,那么将其存储为useReducer使用范围会更广，那么将Context中存储的数据修改为reducer会更加优雅一点，这一部分就留给大家去改造了。

在线实例

另外文中一直尝试的修改数据为同步的，如果有异步的数据修改，是将其在React组件内部处理好，再同步的dispatch出去，还是可以在context组件内部去做更多异步的处理，需要大家自己去实践了。

link-npm包

发表于 2020-10-12 更新于 2020-10-13

可使用 yarn link 或者 npm link

一般用于本地开发，将npm包引入到项目中，而不用发布到npm仓库中，直接进行本地的一个link(实际为一个软连接)，进行引用即可。

引包原理

npm引入包的时候，只要存在于node_modules目录中，（还需要在创建包的目录中添加package.json）就可以在项目中引入到。也就是说只要node_modules中存在符合规则的目录，不需要npm install也可以使用。

可以在一个项目的node_modules中引入一个软连接，这样，应用到的包即为最新的npm包。

在本地创建一个link包

在项目根目录下（与package.json同级的目录）npm link或者yarn link，将该包link到全局包

在本地取消link包

在项目根目录下（与package.json同级的目录）npm unlink或者yarn unlink，将该包取消软连接

引入该link包

在要需要使用项目内进行关联npm link <module-link-name>，之后在该项目中的node_nodules中有了一个软连接，直接指向目标包

之后使用的方式和正常使用方式一样，import <module-link-name>

取消关联

在需要取消的目录中,执行npm unlink <module-link-name>,即可取消该软连接
